Architecture Approaches

How to integrate the Weekly Commits micro-frontend + Java 21 service with Ship

A

Ship as Webpack Host (Full Migration)

Migrate Ship's frontend from Vite → Webpack 5 with Module Federation. Ship becomes the host container. Weekly Commits is a remote federated module with its own Webpack build.

Pros

  • True Module Federation — shared React, shared auth context
  • Single unified build pipeline
  • Future micro-frontends plug in the same way

Cons

  • Vite → Webpack migration is disruptive to Ship
  • Slower dev server (Webpack HMR vs Vite)
  • Higher upfront cost before any feature work begins
B

Webpack Shell Host + Vite Ship (Coexistence)

Create a thin Webpack 5 shell as the host container that loads both Ship (wrapped as a remote) and Weekly Commits (as a remote). Ship keeps Vite internally but exposes itself via a Webpack federation wrapper.

Pros

  • Ship's Vite setup stays untouched
  • Clean separation — shell only handles routing + federation
  • Both remotes are independently deployable

Cons

  • Extra build artifact (the shell)
  • Wrapping Vite output as a federated remote adds complexity
  • Two build systems to maintain
C

Vite Federation (Native Vite)

Use @module-federation/vite (the official Vite plugin for Module Federation). Ship stays on Vite as the host. Weekly Commits uses Vite too, exposed as a federated remote. Same federation protocol, no Webpack needed.

Pros

  • No build system migration — everything stays Vite
  • Fast dev experience preserved
  • Official Module Federation support (not a third-party hack)
  • Lowest integration cost

Cons

  • Vite federation is newer, less battle-tested than Webpack's
  • You specifically asked for Webpack Module Federation
  • Some edge cases with shared dependency version negotiation

💡 Recommendation: Option C (Vite Federation)

Ship is already on Vite. Migrating to Webpack (A) or maintaining two build systems (B) adds significant complexity before you can write a single feature. The @module-federation/vite plugin is maintained by the Module Federation team (same people behind Webpack's implementation) and gives you the same federation protocol. You get shared dependencies, independent deployment, and runtime integration — without the migration tax.
